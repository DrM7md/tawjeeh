# 📋 قواعد العمل (Business Rules)

> هذه القواعد ملزِمة لكل الطبقات (DB constraints + Services + Validation + UI).

---

## A. العام الدراسي (Academic Year)

| # | القاعدة |
|---|---------|
| AY-1 | لا يُسمح بوجود أكثر من **عام دراسي فعّال** واحد (`is_active = true`). تفعيل عام يُلغي تفعيل غيره ذرّيًا (transaction). |
| AY-2 | أي عملية إدخال جديدة تُربط تلقائيًا بالعام الفعّال (`academic_year_id`). |
| AY-3 | عند تغيير العام، **لا تُحذف** البيانات السابقة، بل تُنقل للأرشيف (`status = archived`). |
| AY-4 | يمكن **نسخ التوزيع** السابق إلى عام جديد (مع إمكانية تعديل الموجهين). |
| AY-5 | المؤشرات تعتمد على العام المختار في السياق. |
| AY-6 (مستقبلي) | **تجميد العام**: عند الانتهاء يُمنع التعديل، يُسمح بالاستعراض فقط، وتُنشأ نسخة أرشيف. |

**شاشة الأعوام:** اسم، تاريخ بداية/نهاية، حالة (نشط/مغلق/أرشيف). عمليات: إنشاء، تفعيل (واحد فقط)، إغلاق، أرشفة، نسخ بيانات العام السابق.

## B. الفصل الدراسي (Semester)

| # | القاعدة |
|---|---------|
| SM-1 | لا يوجد فصل نشط خارج عام دراسي نشط. |
| SM-2 | يُسمح بفصل **واحد نشط فقط** داخل العام. |
| SM-3 | العمليات الجديدة ترتبط تلقائيًا بالفصل النشط (`semester_id`). |
| SM-4 | رئيس التوجيه يفتح/يغلق الفصل. |
| SM-5 | عند إغلاق الفصل: يُمنع التعديل، يُسمح بالعرض، تبقى المؤشرات متاحة. |

**حالات الفصل:** لم يبدأ / نشط / منتهي / مغلق. الإعداد الافتراضي: الفصل الأول + الفصل الثاني (قابل للتوسع: فصل صيفي).

**في الترويسة دائمًا:**
```
📚 العام الدراسي: 2026–2027     📖 الفصل الدراسي: الفصل الأول
```
عند التبديل تتغيّر: المؤشرات، الجداول، التقارير، الإحصائيات تلقائيًا.

## C. توزيع المدارس (Distribution)

| # | القاعدة |
|---|---------|
| DS-1 | الهدف: تحقيق **العدالة والتوازن** بين الموجهين. |
| DS-2 | عوامل التوزيع: عدد المدارس، عدد المعلمين، عدد المنسقين، حجم الزيارات المطلوبة، المرحلة. |
| DS-3 | أنماط: تلقائي / يدوي / إعادة توزيع. |
| DS-4 | التوزيع ضمن نطاق القسم (رئيس القسم يوزّع موجهي قسمه فقط). |
| DS-5 | مدرسة+قسم لا تُسند لأكثر من موجه في نفس العام (UNIQUE). |

**مؤشرات:** عدد المدارس لكل موجه، متوسط عدد المعلمين، **نسبة العدالة** (انحراف الحمل عن المتوسط).

**خوارزمية التوزيع التلقائي (مبسّطة):**
1. احسب «وزن» كل مدرسة = دالة(معلمين، منسقين، زيارات مطلوبة، مرحلة).
2. رتّب المدارس تنازليًا بالوزن.
3. وزّعها على الموجهين بأسلوب **least-loaded first** (أقلّهم حملًا أولًا).
4. اعرض النتيجة للمراجعة قبل الحفظ + اسمح بالتعديل اليدوي.

## D. الزيارات (Visits)

| # | القاعدة |
|---|---------|
| VS-1 | نوعان: زيارة **معلم** / زيارة **منسق**. |
| VS-2 | المعلم: زيارة واحدة كحد أدنى خلال العام + زيارات إضافية حسب التصنيف (`required_visits`). |
| VS-3 | التصنيفات: يحتاج دعم / متوسط / متميز (تحدد العدد المطلوب). |
| VS-4 | المنسق: زيارة واحدة سنويًا، ويُسجَّل الفصل الذي تمّت فيه (فلا تظهر كمطلوبة في الفصل الآخر). |
| VS-5 | حالة العرض لكل مستهدف: **تمت ✓ / متبقي ⏳ / متأخر ⚠️** (تُحسب مقابل المطلوب والمواعيد). |
| VS-6 | الزيارات تُسجَّل ضمن العام **والفصل** الفعّال؛ والعرض يفصل الإنجاز بين الفصلين. |

**لوحة المتابعة:** من تمت زيارته / المتبقي / نسبة الإنجاز — قابلة للفلترة حسب العام والفصل والقسم والموجه والمدرسة.

**منطق حالة المعلم (مثال):**
```
required = classification.required_visits   // مثلًا 3
done     = visits(teacher, year).count()
status   = done >= required ? 'تمت'
         : (تجاوز الموعد ? 'متأخر' : 'متبقي')
```

## E. استمارات الزيارة (Visit Forms)

| # | القاعدة |
|---|---------|
| VF-1 | لكل زيارة استمارة واحدة (1:1): بيانات المدرسة، المستهدف، محاور التقييم، الملاحظات، التوصيات، التوقيع، المرفقات. |

| VF-3 | بعد الاعتماد لا تُعدَّل إلا بصلاحية خاصة وتُسجَّل في `audit_logs`. |

## F. تحكيم الاختبارات (Test Reviews)

| # | القاعدة |
|---|---------|
| TR-1 | يُسجَّل أثناء زيارة المدرسة، ويرتبط مباشرة بالعام **والفصل**. |
| TR-2 | المراحل المدعومة: إعدادي (السابع/الثامن/التاسع)، ثانوي، ومشتركة (ابتدائي/إعدادي/ثانوي). |
| TR-3 | التقييم حسب معايير (`criteria` JSON) + رفع نماذج + أرشفة النتائج. |
| TR-4 | تحكيم كل فصل مستقل عن الآخر (إنشاء سجلات منفصلة لكل فصل). |

## G. الاستيراد (Import)

| # | القاعدة |
|---|---------|

## I. عامّة (Cross-cutting)

| # | القاعدة |
|---|---------|
| X-1 | كل عملية حسّاسة تُسجَّل في `audit_logs`. |
| X-2 | الملفات تُخزَّن في تخزين خاص وتُقدَّم عبر مسار محمي بصلاحية. |
| X-3 | الحذف يُفضَّل أن يكون soft-delete للكيانات التشغيلية المهمة. |
| X-4 | كل النطاقات (scoping) تُطبَّق في الـ Backend بغضّ النظر عن الواجهة. |
